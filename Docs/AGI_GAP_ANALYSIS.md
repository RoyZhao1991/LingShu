# 灵枢 × 通用人工智能体验差距评估

> 2026-06-12。对照体验基准：豆包（实时多模态对话的体验天花板）与"贾维斯"（常驻自主中枢的目标形态）。
> 每项标注：本轮已补齐 ✅ / 部分补齐 ◐ / 未动 ✗（附路线图归属）。

## 一、对话体验层（用户每天直接感知的部分）

| 维度 | 豆包/贾维斯的体验 | 灵枢本轮之前 | 现状 |
| --- | --- | --- | --- |
| 直答延迟 | 普通对话 1~2s 开始出字 | 所有消息都套路由 JSON：答案要等完整 JSON 才能解析展示（实测寒暄 3.9s、知识问答 **8.9s**） | ✅ 直答快路：本地分类 → 普通对话跳过 JSON 包装直接流式作答，实测首正文 **~1.6s**、寒暄总耗时 2~4s。误判由「【任务】」升级标记逃生门兜底（实测隐性任务措辞可靠命中） |
| 逐字流式回复 | 回答边生成边上屏 | 执行结果一次性弹出，等待期只有转圈 | ✅ 直答与执行阶段均逐字上屏 |
| 思考过程可见 | "正在想"有真实内容滚动 | M3 的 `<think>` 被整体丢弃，思考期一片空白 | ✅ 思考流实时滚动在气泡推理预览区（尾部 220 字），定稿即清，不进历史 |
| 有限选择交互 | 选项卡一键选择 | 模型把选项铺成长文让用户手打 | ✅ 路由协议新增 `choices`，渲染成选择卡，点选即提交 |
| 语音打断 | 它说话时你开口它就停 | — | ✅ 已有真实 barge-in（电平超阈值 0.25s 停 TTS 重新听），本轮确认非 demo |
| 回复分句流式 TTS | 第一句先念出来 | 整段回复合成后一次性播报 | ✅ 分句早读：流式正文每攒满一句立即排队播报，首句即开口；通话控制器按"播报+队列"判断说话态，分句间隙不误启麦克风 |
| 极简陪伴形态 | 小窗常驻，不抢工作区 | 极简模式占满 1240×820 主窗 | ✅ 收成 340×560 右下角置顶小浮窗：视频画放 + 两条真实电平音轨 |
| 端到端延迟可观测 | — | 无埋点，无从知道 | ✅ 路由/直答/执行三条链路全部带延迟探针（首响/首正文/完成），每轮真实数字进调用链；感知自检覆盖语音/视觉单项延迟 |

实测基线（2026-06-12，MiniMax 官方 M3 直连，含完整路由系统提示）：
路由路径——寒暄 3.9s、知识问答 8.9s（思考 939 字）、任务路由段 26.2s；
直答快路——寒暄 2.1s、知识问答 3.9s（首正文 1.6s）；
路由提示词加入"思考极简"指令后任务路由段思考 1034→298 字。

## 二、记忆层（"它记得我"的感觉）

| 维度 | 目标体验 | 本轮之前 | 现状 |
| --- | --- | --- | --- |
| 语义召回 | "上周让你查的那件事"能命中 | 关键词标签匹配（约 40 个硬编码候选词），换个说法就丢 | ✅ 内嵌 RAG 库：SQLite + FTS5（中文 bigram 切词）+ NLEmbedding 句向量，RRF 混合检索 |
| 旧对话不丢 | 超长对话仍记得开头 | 超预算的旧轮次直接丢弃 | ✅ 上下文压缩引擎：旧轮次折叠成滚动摘要随上下文注入，折叠边界按块推进保前缀缓存 |
| 记忆写入策略 | 存结论而非流水账 | 已有反思摘要雏形 | ◐ 摘要双轨写入（JSON 记录 + 语义库向量化）；模型级"反思"仍是规则压缩（Phase 2 继续） |
| 跨设备/长期容量 | 万条级检索 < 50ms | UserDefaults + JSON 文件 | ◐ 语义库已是 SQLite；任务日志/聊天历史迁移还在 Phase 2 |

关于"内嵌 Redis"的决策：Redis 是独立服务进程，嵌进沙盒 Mac App 意味着进程管理、崩溃恢复、
许可证三重负担，而它的向量检索（RediSearch）在单机单用户场景没有优势。等价能力用
**SQLite（系统内建）+ FTS5 全文索引 + NLEmbedding 本地向量** 实现：零外部依赖、单文件、
进程内调用，且与 ROADMAP Phase 2 的 SQLite 迁移路线同底座。语义库文件：
`~/Library/Application Support/LingShu/Memory/semantic-memory.sqlite3`。

## 二点五、情境智能层（2026-06-12 第三轮新增）

| 维度 | 要求 | 现状 |
| --- | --- | --- |
| 声线判断说话人 | 听出男女、调整语气 | ✅ `LingShuSpeakerProfiler`：自相关基频估计 + 滚动中位数 → 男声/女声/未定（带置信度），画像注入对话上下文；语气怎么调由模型决定，非写死策略。实测合成 120Hz/220Hz 音准确分类 |
| 情境化回答 | 深夜问"几点了"该提醒休息；说要演示该结合摄像头打趣 | ✅ 拆除写死的本地时间/寒暄秒回；`LingShuSituationContext` 常驻注入（时间/时段/连续使用时长/声线画像/视觉摘要/后台任务进展）。真实 M3 实测：深夜 01:47 问时间 →"已经凌晨了，注意休息 🌙"；下午对照组不唠叨；摄像头见 2 人 →"是您的同事已经在旁边了吗" |
| 像朋友一样互动建议 | 主动、具体、可拒绝 | ✅ 人格提示加入分寸感约束（"点到为止，不要每句都提"），建议内容由模型按情境生成 |

## 三、任务与自主性层

| 维度 | 贾维斯形态 | 灵枢现状 | 归属 |
| --- | --- | --- | --- |
| 规划-执行-验证闭环 | 给目标自动拆解、失败自修 | ✅ 协同管线：规划→专家产出→评审→纠正→验收五阶段真实模型调用，评审"需修正"自动打回修订一轮，全程进任务记录（不再是每个 agent 一句台词的剧场） | 本轮落地（Phase 5 核心） |
| 专家知识库与模板 | 项目经理出项目分析、产品出 PRD、架构师出架构文档 | ✅ `LingShuExpertProfileRegistry`：五类专家档案（知识要点+交付模板+评审清单），协议化可插拔 | 本轮落地 |
| 任务上下文管理 | 迭代继承、顺序执行、任务间隔离 | ✅ 管线调用不携带聊天历史（严格隔离）；同任务迭代经执行记录血缘继承前序结论；同线程队列顺序执行 | 本轮加固 |
| 资源准入 | 系统忙时排队并告知 | ✅ `LingShuTaskAdmissionPolicy`：在跑管线数×CPU 负载×内存余量综合评定，排队原因明示用户，环境合适自动推进 | 本轮落地 |
| 主动汇报 | 完成后灵枢主动发起 | ✅ 管线收口输出结构化汇报（交付/过程/验收结论），由灵枢主动发到对话 | 本轮落地 |
| 执行期互动 | 边干活边聊 | ✅ 非任务消息走并发直答（独立句柄，不碰执行状态机），情境上下文带着任务进展 | 本轮落地 |
| 真实工具执行 | 能动文件、查资料、跑代码 | API 通道只产出文本；本地 builder 兜底生成文档 | Phase 5 剩余（能力节点插件化） |
| 常驻与主动开口 | 7×24 + 基于时间/事件主动说话 | 关窗即退，无触发器系统 | Phase 6 |
| 产出物可见性 | 交付物一目了然 | ✅ 任务详情右侧产出文件面板（预览/打开/访达，含续接历史产物） |

## 五、模型通道层

- ✅ 回复适配器模式落地：`LingShuModelReplyAdapting`（最终文本归一）+ `LingShuReplyStreamParsing`
  （流式增量归一，处理跨 chunk 标签切碎）。M3/Qwen/DeepSeek/GLM/Kimi 走内联 think 适配器，
  标准 OpenAI 兼容走直通适配器。新模型族只在 `LingShuModelReplyAdapters` 登记一行，
  路由/执行/展示零改动——切模型即插即用。
- ✗ 常驻双工通道（每次仍是独立 HTTP 流式请求）：Phase 3。
- ✗ 按任务特征分级路由（快通道小模型判断 + 强模型执行）：Phase 3。

## 六、一句话结论

本轮把"对话体验层"和"记忆层"推到了接近豆包的水准（流式、思考可见、选择卡、小窗通话、
语义记忆、上下文不丢）；与"贾维斯"的本质差距集中在 **Phase 5 自主任务闭环**（规划-执行-验证）
和 **Phase 6 常驻主动性**——这两块是接下来唯一值得投入的主线。
