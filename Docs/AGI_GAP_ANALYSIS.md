## 二、记忆层（"它记得我"的感觉）

| 维度 | 目标体验 | 本轮之前 | 现状 |
| --- | --- | --- | --- |
| 语义召回 | "上周让你查的那件事"能命中 | 关键词标签匹配（约 40 个硬编码候选词），换个说法就丢 | ✅ 内嵌 RAG 库：SQLite + FTS5（中文 bigram 切词）+ NLEmbedding 句向量，RRF 混合检索 |
| 旧对话不丢 | 超长对话仍记得开头 | 超预算的旧轮次直接丢弃 | ✅ 上下文压缩引擎：旧轮次折叠成滚动摘要随上下文注入，折叠边界按块推进保前缀缓存 |
| 记忆写入策略 | 存结论而非流水账 | 已有反思摘要雏形 | ◐ 摘要双轨写入（JSON 记录 + 语义库向量化）；模型级"反思"仍是规则压缩（Phase 2 继续） |
| 跨设备/长期容量 | 万条级检索 < 50ms | UserDefaults + JSON 文件 | ◐ 语义库已是 SQLite；任务日志/聊天历史迁移还在 Phase 2 |

关于"内嵌 Redis"的决策：Redis 是独立服务进程，嵌进沙盒 Mac App 意味着进程管理、崩溃恢复、
许可证三重负担，而它的向量检索（RediSearch）在单机单用户场景没有优势。等价能力用
**SQLite（系统内建）+ FTS5 全文索引 + NLEmbedding 本地向量** 实现：零外部依赖、单文件、
进程内调用，且与 ROADMAP Phase 2 的 SQLite 迁移路线同底座。语义库文件：
`~/Library/Application Support/LingShu/Memory/semantic-memory.sqlite3`。

## 五、模型通道层

- ✅ 回复适配器模式落地：`LingShuModelReplyAdapting`（最终文本归一）+ `LingShuReplyStreamParsing`
  （流式增量归一，处理跨 chunk 标签切碎）。M3/Qwen/DeepSeek/GLM/Kimi 走内联 think 适配器，
  标准 OpenAI 兼容走直通适配器。新模型族只在 `LingShuModelReplyAdapters` 登记一行，
  路由/执行/展示零改动——切模型即插即用。
- ✗ 常驻双工通道（每次仍是独立 HTTP 流式请求）：Phase 3。
- ✗ 按任务特征分级路由（快通道小模型判断 + 强模型执行）：Phase 3。

## 六、一句话结论

本轮把"对话体验层"和"记忆层"推到了接近豆包的水准（流式、思考可见、选择卡、小窗通话、
语义记忆、上下文不丢）；与"贾维斯"的本质差距集中在 **Phase 5 自主任务闭环**（规划-执行-验证）
和 **Phase 6 常驻主动性**——这两块是接下来唯一值得投入的主线。
