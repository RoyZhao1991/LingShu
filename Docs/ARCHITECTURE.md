# LingShu Architecture Standard

灵枢是通用 agent 能力中枢，不是单一开发工具。代码结构必须服务于长期扩展：实时语音、视觉解析、外部 agent、多个模型网关、任务记忆和执行审计都应能独立演进。

## Core Principle

灵枢主线程只做判断、记忆恢复、能力编排、权限裁决、验收和统一回复。具体工作由内部或外部能力节点完成。

任何新能力都必须先回答三个问题：

1. 它属于用户界面、应用编排、领域规则、基础设施适配，还是外部服务？
2. 它是否会让主线程直接承担具体执行工作？
3. 它产生的过程和产物是否能被任务执行记录追溯？

## Layer Boundaries

### App

路径：`Sources/LingShuMac.swift`

只允许放 App 启动、窗口初始化和根视图挂载。不得放业务模型、模型调用、agent 调度或大块 UI 组件。

### Domain

路径：`Sources/Domain/`

放稳定领域对象和枚举，例如消息、agent 状态、任务阶段、记忆记录、模型配置描述。Domain 不得调用网络、文件系统、Codex CLI 或 SwiftUI 页面逻辑。

允许依赖：`Foundation`，必要时允许轻量 UI 值对象，例如 `Color`。  
禁止依赖：具体模型 SDK、Codex、外部 agent HTTP 客户端、视图状态。

### State

路径：`Sources/State/`

放界面可观察状态和主线程编排状态。State 可以组合服务，但不应直接实现底层模型协议、文件持久化细节或外部 agent 协议。

规则：

1. 主线程状态必须常驻，负责心跳和最近会话恢复。
2. 每条用户消息都对应一个任务执行记录。
3. 主线程先判断是否直接回答，再决定是否创建任务线程或调用 agent。
4. 普通问答也要经过记忆检索，但不一定创建子线程。

### Application Services

路径：`Sources/Services/`

放语音、视觉、任务生命周期、执行记录聚合等应用级服务。服务可以被 State 调用，但服务之间不得形成循环依赖。

新增实时语音能力时，入口应放在 `Services/Voice` 或现有 `VoiceIOManager` 的后续拆分文件中。新增视觉解析能力时，入口应放在 `Services/Vision`，模型适配走 `Infrastructure/ModelGateway`。

实时感知能力必须分三层：

1. 设备采集层：麦克风、音箱、摄像头，只负责采集和播放，不直接决定业务流程。
2. 本地解析层：Speech、Vision、TTS 等系统能力，用于低延迟兜底和权限内观察。
3. 实时感知网关：统一封装音频包、视频帧、转写文本、视觉摘要和语音输出事件，按路由转发给本地解析、实时大模型或外部适配器。

设备层不得硬编码任何模型供应商。新增 GPT、DeepSeek、MiniMax、Claude、自托管视觉/音频模型时，必须通过 `LingShuRealtimePerceptionGateway` 或其后续拆分出来的 provider 注册，不得直接改麦克风或摄像头服务。

认主流程属于实时感知网关后的身份锁能力，而不是麦克风、摄像头或某个模型供应商的私有逻辑。开启认主后，灵枢必须同时采集面容和声线样本，形成可替换的主人身份档案；身份锁开启后，触发词命中也必须先通过面容和声线联合确认，才能进入实时对话或自动执行任务。当前本地实现只作为可演示的轻量特征锁，后续 H100、云端声纹、人脸识别或多模态模型只能替换特征提取和校验适配层，不能改动主线程对话流程。

默认身份档案位置：`~/Library/Application Support/LingShu/Identity/owner-profile.json`。档案只保存面容/声线特征向量和阈值，不保存原始音频或视频帧。

### Infrastructure

路径：`Sources/Infrastructure/`

放外部世界适配，包括 Codex CLI、OpenAI/DeepSeek/Claude/MiniMax 等模型网关、外部 agent 协议、文件系统仓储、网络客户端。

规则：

1. Infrastructure 不决定产品流程，只提供能力。
2. 模型供应商差异必须封装在适配层，不得散落在 UI 或 State 中。
3. 长连接、流式输出、心跳、自恢复和重连策略属于模型网关或远端会话池，不属于视图。
4. 内部诊断日志不得直接进入用户对话，只能进入执行记录或调试窗。
5. 实时音视频模型如果需要 WebSocket、SSE、gRPC 或厂商 SDK，应在 Infrastructure 增加 provider/adapter，并对外暴露统一的实时感知事件合同。
6. 新增 Kimi、DeepSeek、Qwen、MiniMax、本地 vLLM 等主模型通道时，只能通过模型网关适配层声明接口、协议和能力；不得让业务流程依赖具体模型名称。

### Views

路径：`Sources/Views/`

只放 SwiftUI 展示和交互。Views 可以读取 `LingShuState`，但不得直接调用 `CodexBridge`、网络客户端、文件仓储或外部 agent。

规则：

1. 新页面按功能建文件，不允许继续堆回主文件。
2. 复杂组件超过约 300 行时必须继续拆分。
3. 视图文案必须呈现真实状态，避免“已就位”这类无信息量文案。
4. 聊天主界面只展示灵枢统一人格；agent 过程进入调用链或任务执行记录。

### Support

路径：`Sources/Support/`

放通用扩展、颜色、格式化、小工具。不得放业务流程。

### Tests

路径：`Tests/`

新增规则或调度行为必须配测试。优先测试纯函数和服务边界：

1. 意图判断和任务路由。
2. 记忆命中、压缩、冷备回溯。
3. agent 调度顺序和权限边界。
4. 模型网关错误映射、流式心跳、重连策略。
5. 任务执行记录和产物清单。

macOS 可能拒绝直接加载 SwiftPM 在 `.build` 下生成的 XCTest bundle。项目测试入口统一使用 `Scripts/run-tests.sh`：脚本会先构建测试包，再复制到临时目录、清理扩展属性、重签名并调用 XCTest。后续不要把 `swift test` 卡住误判为测试逻辑失败。

## Memory Standard

灵枢采用两层记忆：

1. 主线程记忆：用于判断当前消息是否延续已有主题、是否需要加载历史线程、是否需要创建子线程。
2. 执行记忆：用于恢复某个任务线程的执行过程、产物、错误、验收结果和下一步建议。

记忆过长时必须摘要压缩。长期未访问的记忆进入冷备库。冷备可以被检索，但不得默认全量加载。

## Task Journal Standard

每条用户消息必须创建任务执行记录。记录内容以“群聊式进度”呈现，包括：

1. 用户原始请求。
2. 灵枢主线程判断。
3. 记忆检索结果。
4. 模型网关状态。
5. 被调用 agent 的发言式进度。
6. 产出物清单。
7. Review 和交付结论。

如果新任务续接历史任务，执行记录必须能关联并展示历史任务链路。

## Chat History Standard

聊天气泡是用户感知灵枢连续性的第一入口，必须独立持久化，不能只依赖任务执行记录或主线程摘要。

规则：

1. `chatMessages` 的新增、更新和删除必须进入聊天历史存储。
2. 默认热加载最近 3 天的聊天历史，保证 App 重启后对话连续。
3. 超过热加载窗口的消息进入冷备库，不在启动时全量加载。
4. 用户向上翻历史时，界面应自动分页拉取冷备消息；冷备加载流程对用户透明。
5. 加载中的临时气泡不得持久化为正式历史，避免重启后出现永远转圈的回复。
6. 语音输入、视觉触发、插件消息和文字输入最终都必须落成同一种 `ChatMessage`，再复用统一存储链路。

默认文件位置：`~/Library/Application Support/LingShu/History/chat-hot.json` 和 `~/Library/Application Support/LingShu/History/chat-cold.json`。后续如果历史量继续增长，可以在不改变界面调用方式的前提下迁移到 SQLite。

## Design Delivery Standard

设计部是通用设计交付能力，不是 PPT 专用脚本。它负责把用户目标转成受众判断、叙事结构、视觉方向、页级版式、图表建议和交付验收口径。

当用户提出 PPT、演示文稿、汇报材料、版式、视觉方案、设计稿等请求时，灵枢必须先走主线程判断，再按需创建设计交付任务。复杂交付必须至少经过规划、审议、调度和设计节点；如果需要真实文件，应继续调度执行节点生成文件，并由验证节点检查可打开性、页数、版式完整性和文件路径。

设计部不得直接耦合某个幻灯片库、模板或模型供应商。PPTX、HTML 演示页、图片素材、图表和外部设计 agent 都应通过执行器或外部 agent 网关接入，并把产物登记到任务执行记录的产出物清单。

## Agent Standard

灵枢能力节点必须是通用能力，不允许写死为某个一次性任务。推荐能力类型：

1. 规划
2. 执行
3. 监控
4. 验证
5. 记忆
6. 安全
7. 知识
8. 设计交付
9. 语音交互
10. 视觉解析
11. 实时感知路由

外部 agent 接入必须通过注册表和网关，不得在主线程中硬编码远端地址和协议细节。

## File Size Guideline

1. App 入口文件目标小于 200 行。
2. 单个 SwiftUI 页面目标小于 300 行。
3. 单个服务文件目标小于 500 行。
4. 超过 800 行必须拆分，除非是明确的数据目录或生成代码。
5. `LingShuState` 只是过渡性聚合点，后续应继续拆成更小的 store 和 coordinator。

## Review Checklist

每次迭代完成前检查：

1. 主文件是否仍只承担启动职责。
2. UI 是否直接调用底层模型或系统命令。
3. 新能力是否有明确层级归属。
4. 普通问答和复杂任务是否都有任务记录。
5. agent 状态是否只展示当前任务真实参与者。
6. 模型断线、重连、心跳是否有用户可理解状态。
7. 构建和核心测试是否通过。
