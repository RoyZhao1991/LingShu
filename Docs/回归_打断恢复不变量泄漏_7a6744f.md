## 一句话

在 `.classic` 引擎下,**飞行中打断(`lingshu_stop`)→ 同一持久主会话发下一条干净任务**,会让进程级循环不变量计数器 `loopInvariantViolations` **持续爬升**(实测 0→3→8),回合虽仍到终态(不 wedge),但不变量被违反 = robustness 网在响,根因未除。

## 复现(我已连复两次)

```bash
# 当前 commit:7a6744f(完全版),真模型 DeepSeek,.app 经 8917 驱动
LINGSHU_SOAK_TASKS=6 LINGSHU_SOAK_MINUTES=20 bash Scripts/soak-e2e.sh
```

打断恢复段逻辑(`Scripts/soak-e2e.sh` 第 186–198 行):发一条"写素数筛+测一下"的代码任务 → `sleep 7`(飞行中)→ `lingshu_stop`(中途打断)→ `sleep 4` → 发干净任务"3 加 4 等于几" → 断言 `invariants()==baseline`。

## 期望

打断恢复段三轮 `不变量` 恒为 `baseline`(0);干净任务到终态且不留账。

## 根因分析(独立验收判断,非文档自述)

1. **真凶 = 飞行中取消留孤儿**:`lingshu_stop` 取消正在飞的回合(`activeAgentTurnTask` / `cancelAllRunning`)。取消点若落在"assistant 已产出 `tool_calls`、对应 `tool` 结果尚未回填"之间,**持久主会话的 history 会留下未应答的 tool_call**。下一回合 `driveAgentDelivery → session.send` 在回合边界 `checkInvariants` 命中 I1/I2(孤儿 / 未应答 tool_call)→ 记账到 `LingShuLoopInvariantTelemetry`。
2. **为何不 wedge / 不 400**:网关侧 `LingShuGatewayAgentModel.sanitizeToolCallSequence` 在序列化前补占位/丢孤儿 → 不发 400、能到终态。**但会话层的不变量检查发生在 sanitize 之前**,违反照记 → 计数器爬升。即"网关兜底闸"掩盖了"会话层留孤儿"这个根因,robustness 网如实报了出来。
3. **`driveAgentDelivery` 入口的 `batchInterruptRequested=false` 复位治不了这个**:它治的是"打断标志泄漏旁路验收门",不治"history 里已存在的孤儿 tool_call"。

### 自主段 FAIL 是级联误报,不是独立 bug(重要校正)

soak 自主段断言 `ok = on and health_ok() and (invariants()==baseline)`,而 `baseline` 是**开跑时的 0**、`invariants()` 是**进程累计**、**段间从不复位**。打断段已泄漏 8 之后,后续每个 `invariants()==baseline` 必为假 → go_live/在岗被判 FAIL,**即便上岗本身成功**(打印 `standingPersonOnDuty=True` 已证明上岗成了)。

证据:旧树 `soak5h.log`(打断零泄漏)自主段全 PASS;凡打断泄漏的 run 自主段才 FAIL。**根因收敛为一个:打断恢复的不变量泄漏。**

## 两处待修

1. **【主】会话层**:飞行中取消后,在下一回合 `send`/`resume` 前**修齐 history 的孤儿 tool_call**(补合成结果或剪除),让会话层与网关 sanitize 同口径 —— 把不变量违反消灭在记账之前,而非靠网关兜底掩盖。建议落点:`LingShuAgentSession` 取消恢复路径 / `driveAgentDelivery` 入口(对标 `sanitizeToolCallSequence` 的会话层版本)。
2. **【次·测量】soak 脚本**:自主段应**按段重新取基线**(或断言"本段增量==0"),避免上一段的泄漏把无辜的下一段也染红,掩盖真实定位。
