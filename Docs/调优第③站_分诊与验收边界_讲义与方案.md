## Part A 讲解:这一站要真正学会的东西

### A0. 全景:你追的"分诊 bug",病根其实不在分诊

一个回合的决策链(简化):

```
submitTextInput
  → ② 确定性快路径(显式 @ 协议)
  → ③ classifyDispatch 归属解析(Triage.swift:39)     ← 只判 .reply / .chat
  → ④ deriveGoalSpec(GoalSpec.swift:27)              ← 模型 1-shot 判 kind: task/interaction/question
  → ⑤⑥ 主 agent 循环(模型即编排者)
  → ⑧ verifyAndContinue / runCompletionGate           ← 谁该验收、谁直接放行
```

**①已知事实**:重构后的 [Triage.swift](../Sources/State/LingShuState+Triage.swift) 已经"瘦身"成**归属解析器**——只回答"这条输入是不是在回复某条等待中的线程",不再判 chat/task、不创建子任务(文件头注释即此约定)。这个方向是对的,符合"模型即编排者"铁律。

**①已知事实**:而"纯问答被卷进任务验收"的病根在 [CompletionGate.swift:91](../Sources/State/LingShuState+CompletionGate.swift):

```swift
let taskLike = taskLikeBase || replyNeedsUser || replyNeedsExecutionInput
```

`replyNeedsUser` / `replyNeedsExecutionInput` 是对**回复文本**的关键词判定,它们绕过了 `taskLikeBase`(:76,结构化判据),让一条 `goalSpec.kind == .question` 的纯问答仅凭回复措辞就进入验收/待用户。而 `replyRequestsUserExecutionInput`(:463)自己的注释写着"调用方必须先确认这是 task/interaction 记录"——**注释承诺的前置条件,代码没有兑现**。

> **本站第一课(架构视角)**:③ 把"隐形分诊"从入口删掉了,⑧ 又用关键词把它加了回来——**决策拓扑(decision topology)漂移**。判定"这个回合是不是任务"的职责,必须锚定在一个地方(④ 的 GoalSpec + 结构事实),不能散落在各层用词形重判。

### A1. [新]⭐ 用 LLM 做分类(classification)

分类是最小的 LLM 任务形态:**单轮、无工具、输出枚举**。灵枢里有三个分类器,档次不同:

| 分类器 | 位置 | 手段 | 档次 |
|---|---|---|---|
| 归属解析 | Triage.swift:39 `classifyDispatch` | 模型 1-shot,输出一行 JSON | 概率,但输入是完整上下文 |
| GoalSpec.kind | GoalSpec.swift:27 `deriveGoalSpec` | 模型 1-shot,三选一枚举 | 概率,专职判 task/interaction/question |
| 回复词形判定 | CompletionGate.swift:437/:463 | 关键词表 contains | **穷人版分类器** |

关键词表的失败模式是**双向增长**的:每加一个词,误伤面和漏判面同时变大。本次实诊的两个活教材(**①已知事实**,由本轮新测试与词表逐条推演坐实):

- **反例 A(误伤)**:问"什么是闭包?",模型正常收尾"……如果你想深入了解,请**告诉我**,我可以**继续**展开" → `replyRequestsUserExecutionInput` 命中("告诉我"+"继续")→ 纯问答变"补充信息"选择卡。
- **反例 B(误伤,更刺眼)**:问"什么是 OAuth?",回复含"token"+"第三方" → `replyRequestsUserPrerequisite` 命中 → 百科式回答变**授权卡**。这个旧函数以前对 chat 记录永远够不着(taskLike guard 早退),:91 的绕过把它放了出来。
- **漏判(同一词表的另一面)**:真任务回复"第一步**已完成**。请告诉我目标库,我才能继续"——因含豁免词"已完成"直接放行,该拦不拦。

> **本站第二课**:**分类交给模型或结构化事实;关键词只能做保守护栏(宁可漏、不可抢)**。护栏的合法用途是"挡明显不该劫持的"(如 Triage 里的附件保护),不是"抢着做正向判定"。

### A2. [新]⭐ structured output:结构是"保证出来的"还是"清洗兜出来的"

归属解析要求模型输出一行 JSON,当前靠 `stripThinkTags` + 字符串匹配解析(Triage.swift:287 `parseContextResolverDecision`)。**①已知事实**:抓手已埋好——[LingShuControlPlaneBaseline.isDirtyJSON](../Sources/Support/LingShuControlPlaneBaseline.swift)(:29)在每次调用后量"脏率"(模型直接吐纯 JSON,还是裹着 think/解释文字被清洗兜出来),trace 落 `triage-baseline.log`(sentinel 文件门控)。

学习点:结构约束的三档强度——**prompt 约定(现状)< JSON mode < tool-call schema**。脏率是决定要不要升档的依据,先量后动。

### A3. [新]⭐ 小而快的分类 prompt + "值不值调一次模型"

**①已知事实**,归属解析已经把"值不值"做对了三处(Triage.swift):

1. **免费路径优先**:无候选待答线程 → 不调模型直接交主脑(:65-69);有候选但词形保护全不匹配 → 同样不调(:60-64)。
2. **失败姿势正确**:模型超时/未完成 → `brainFailed`,**不劫持输入**、交主脑按普通输入处理(:103-107)。分诊挂了绝不能挡住主脑——fail-open。
3. **可观测**:每次调用记 耗时/载荷字符/落在哪个脑(:100-102),回答"这跳值不值"。

### A4. ⑧站前哨:判定手段的光谱

验收门可用的判定手段,按可靠度排序:

```
确定性结构事实                LLM 1-shot 判定              回复词形嗅探
(goalSpec.kind、artifacts、   (GoalSpec 同款模式,          (contains 关键词表)
 ask_user→.blocked、           一次轻调用)
 工具调用记录、测试绿)
        ▲ 首选                    ▲ 结构事实缺位时的兜底        ▲ 只配当保守护栏
```

**①已知事实**:结构化信号在本仓库全部已存在——`recordNeedsCompletionGate`(CompletionGate.swift:502,五条结构判据)、`goalSpec.kind`(模型判)、agent 循环的 `ask_user → .blocked` 一等语义。**验收门的输入应该是 typed 事实,不是回复的词形。** 用词形做门还有一层古德哈特问题:词表一旦成为门,模型措辞稍变(被 system prompt 引导、换模型)门就失效,而你以为它还在。

### A5. 顺带看清的两件事(与 ③ 相邻)

- **队列现象是下游症状**(**①已知事实**):串行队列一次只放一条、排队项出队才建记录([SerialInputQueue.swift:90-92](../Sources/State/LingShuState+SerialInputQueue.swift))。第一条纯问答被拖进验收循环(verifyCeiling=8 轮、无全局超时)后耗时暴涨 → 后 3 条永远排队无记录 → 用户看到"只有 1 条记录长期执行中"。**修好 ⑧ 的误入,队列现象大概率自消**。
- **观察收口开了反向旁路**(**①已知事实**,已用冒烟用例原话验证):`readOnlyObservationDeliveryCanFinish`(CompletionGate.swift:141)的变更词表(:172)只有"写入/写一个",没有裸"写/实现/编写/修复/改";smoke-e2e 用例 1 原话"在目录 X **写** add.py……最后一句话**告诉我**结果与文件路径"会命中观察意图、躲过变更意图 → 交付型编码任务可借 fast-path 同时跳过 maker≠checker 验收([AgentAcceptance.swift:135](../Sources/State/LingShuState+AgentAcceptance.swift))和成功标准核验。**②经验性猜测**:GoalSpec objective 文本有时含"生成/创建"能救回,但不能指望。

## Part B 解决方案说明(最优解收口,按"只改一个变量"排期)

> 总原则:把"进不进验收"的裁决权收回**结构化信号**(goalSpec.kind + recordNeedsCompletionGate + ask_user 语义);回复词形判定降级为 taskLike 记录内部的兜底,永不对 .question/.chat 生效。
> 约束:脏工作区,只碰本问题相关文件(CompletionGate / AgentAcceptance / SerialInputQueue / Triage / 对应测试 / soak 脚本)。

### 步骤 0(动手前):量基线

- 开 sentinel:`touch ~/Library/Application\ Support/LingShu/.baseline-log-enabled`。
- 跑 `Scripts/jarvis-soak.sh` 的 `queue_probe`(4 连发唯一 token,本轮 WIP 已建好的回归网)+ 手动 4 连发纯问答。
- 记录基线三数:**纯问答误入验收次数 / 4 连发全部到终态耗时 / 归属解析脏率与耗时分布**(triage-baseline.log)。

### 步骤 1(P0,根治误伤):恢复 taskLikeBase 前置

- **假设**:反例 A/B 类误伤全部来自 :91 的绕过;收回后纯问答 4 连发全部"已直接回答"。
- **改动(一个变量)**:[CompletionGate.swift:91](../Sources/State/LingShuState+CompletionGate.swift) 改为——`replyNeedsUser`/`replyNeedsExecutionInput` 仅当 `taskLikeBase == true`,**或** goalSpec 缺失/`kind == .unknown`(真正的"漏分类"兜底)时参与裁决;`kind == .question` 显式豁免。:97 快速放行与 :122-125 的强灌同步收敛。
- **测试守门**:
  - 改写 `testExecutionInputRequestIsCaughtEvenWhenRecordWasNotClassifiedAsTask`(TaskCompletionGateTests.swift:136)——它现在守的是**错误不变量**;改成 goalSpec 缺失/unknown 场景下仍拦得住。
  - 新增两个负例:反例 A("……请告诉我,我可以继续展开")与反例 B(OAuth 科普含 token/第三方),`.question` 记录必须 `.ok`。
- **量**:重跑步骤 0 三数。**回退条件**:soak 中"任务型回合文本问句漏拦"上升(观察 waitingForUser 覆盖率)。

### 步骤 2(P1,机制侧根治"模型用文本问问题而非 ask_user")

- **推荐方案**:task 记录上,回复疑似在向用户提问时,**用控制面模型 1-shot 判定**"这条回复是否在向用户索取继续执行所需的输入"(与 deriveGoalSpec 同模式,复用 `controlPlaneModelAdapter` + LingShuControlPlaneBaseline 观测);现有关键词表降级为"值不值花这一跳"的预筛。
- **配套(更长期)**:task 记录系统提示强化"要问用户必须走 ask_user 工具"(循环已有 .blocked 一等语义);遥测统计"文本问句"发生率,趋零后删兜底——**兜底应当有退役日期**。
- 步骤 1 量完再动这里,不并轮。

### 步骤 3(P1,堵观察收口旁路)

- **改动**:`readOnlyObservationDeliveryCanFinish` 加结构闸——`goalSpec.kind == .task && !successCriteria.isEmpty` 时不允许 fast-path;变更词表补裸"写/实现/编写/修复/改/做一个"。
- **测试守门**:把 smoke 用例 1 原话("写 add.py……告诉我结果与文件路径")作为负例进 TaskCompletionGateTests——现负例"生成一个报告文件"恰好选了表内词,掩盖了覆盖不足。

### 步骤 4(P2,队列韧性)

- `LingShuPendingSerialInput` 补 `imageDataURLs`(对称 [DispatchQueue.swift:21](../Sources/State/LingShuState+DispatchQueue.swift) 的队列项;现状 `pendingDirectBrainImages` 单槽 consume 即清,忙时连发带图输入会丢图/串图)。SerialInputQueueTests 补带图排队用例。
- 主回合加 watchdog/执行预算(第一回合悬死时队列唯一出路不再是手动停止;`cancelCurrentCall` 同步释放闸门本轮已修好且有测试)。

### 步骤 5(P2,状态可见性,直接消掉问题清单 #8)

- missionStatus 显示"验收 第 N/8 轮";任务卡状态区分 **排队 / 执行 / 验收中 / 待用户**(现在 UI 只显示 status rawValue,验收轮次与等待原因不可见)。

### 遗留小项(顺手,不占独立轮次)

- hub-soak `await_record_for_anchor` 的 `before_ids` 死参数(jarvis 版有用,hub 版没接)。
- 演示答疑残洞:[LingShuInteractionFulfillment.isHollowInteractionStatus](../Sources/Support/LingShuInteractionFulfillment.swift)(:106)对"空心词+实质词共现"的混合文本判非空心("已完成答疑。……因为第二页……"会被念出);placeholders 表缺"已完成答疑"本身;演示答疑**内容真实性**无自动断言(现有测试只测播放头位置)。属演示链路,独立于 ⑧,可另开一轮。

### 第③站的收尾定义(什么时候能在大纲打钩)

1. 步骤 1 落地:反例 A/B 负例绿 + 改写后的兜底测试绿 + 全量 `swift test` 绿;

## 一页记忆卡(本站带走的四句话)

1. **回合是不是任务,只能判一次**——判在 ④(GoalSpec),⑧ 只消费 typed 事实,任何层不得用词形重判。
2. **关键词表是穷人版分类器**:只配当保守护栏(宁可漏、不可抢),且每张护栏都该有退役日期。
3. **分类调用的三件套**:免费路径优先、fail-open 不劫持、每跳可观测(耗时/载荷/脏率)。
4. **测试绿 ≠ 不变量对**:`testExecutionInputRequestIsCaughtEvenWhenRecordWasNotClassifiedAsTask` 全绿地守护着一个 bug——写测试前先问"我在固化哪条边界"。
